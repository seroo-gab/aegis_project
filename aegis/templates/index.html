Merkle Inclusion Proof
📋 What is a Merkle Inclusion Proof?
Think of it like a receipt for a single sensor reading.
Every 10 minutes, the gateway takes all accepted readings and creates a single summary fingerprint (called a Merkle root) that gets published to the Solana blockchain.

An inclusion proof answers the question: "Can I prove that this one specific reading was part of that blockchain snapshot — without downloading every other reading?"

Yes. The proof provides a short mathematical path from a single record up to the blockchain-anchored root. Anyone can verify this path independently without needing the full dataset. It's useful when a node owner only has their own readings and wants to prove they were recorded correctly.

How AEGIS Works — Plain Language Guide
Everything you need to know about connecting sensors and understanding what the gateway does.
What is AEGIS?
AEGIS is a security layer for IoT sensor data. It runs on a Raspberry Pi and sits between your ESP32 sensor boards and any downstream system that uses the data. Its job is simple: make sure every reading is real, on time, and has not been altered. Once a reading passes, it gets a permanent digital fingerprint. Every 10 minutes, a summary of all fingerprints is uploaded to the Solana blockchain — a public ledger that nobody can change — so that you can always prove the data is genuine.
📋
Step 1 — Register your sensor
Before a sensor can send data, you register it in the system. You tell the gateway its name, how often it will send readings, and what kind of values to expect. Any unknown device trying to send data is silently ignored.
🔍
Step 2 — Every reading is checked
Each reading that arrives gets three automatic checks: Was it sent on time? Is it the next expected reading in the sequence (no skips, no duplicates)? Is the sensor connection still live? If any check fails, the reading is rejected and the reason is logged.
🔗
Step 3 — Data is fingerprinted and sealed
Every accepted reading gets a unique fingerprint that includes the previous reading's fingerprint — like a chain of wax seals. Every 10 minutes, one combined fingerprint for the whole batch is published to the Solana blockchain for permanent, public proof.

What does a sensor reading look like?
Every time a sensor takes a reading, it packages it into a small text message — a JSON object — and sends it to the gateway. Think of it like a short form the sensor fills out for every reading it takes. The form has exactly four required fields.
⚠ All four fields must be present. A message missing any of them is thrown out immediately and logged as MALFORMED_PACKET.
{
  "node_id": "Indoor_Temp_Sensor_1",  // The sensor's registered name
  "seq_num": 1,  // A counter that goes up by 1 with each reading
  "timestamp": 1735084800000,  // The exact time the reading was taken (milliseconds since 1970)
  "sensor_value": 28.50,  // The primary measurement — temperature in °C for DHT11
  "humidity": 72.00  // Extra field — ignored by enforcement, stored for reference
}
node_id
The sensor's name. Must exactly match the name you registered in the Node Enrollment page — even one wrong letter will cause the reading to be rejected as UNENROLLED_NODE. Current nodes: Indoor_Temp_Sensor_1, Indoor_Temp_Sensor_2, Indoor_Temp_Sensor_3.
seq_num
A simple counter that increases by exactly 1 with every reading. Reading 1, then 2, then 3 — no skipping, no going back. The gateway uses this to detect missing or duplicate readings. If a board restarts, the counter resets to 1 — you must do a Session Reset in Operator Tools before it can send again.
timestamp
The time the sensor took the reading, in milliseconds since 1970. The gateway compares this against its own clock — if the clocks are more than 3 seconds apart, the reading is rejected. The board must sync its clock via NTP before sending its first packet.
The humidity field is extra — and that is fine. All three DHT11 sensors send both "sensor_value" (temperature in °C) and "humidity" (relative humidity) in every packet. The gateway only runs enforcement checks on the four required fields. The humidity value is stored in the raw packet record but is not verified, hashed, or anomaly-checked. If you add more extra fields, they are also just stored and ignored by enforcement.
How does the sensor connect to the gateway?
The sensor connects to the gateway using a protocol called MQTT — a lightweight messaging system designed for small devices. Think of it like a radio channel: the sensor broadcasts on a specific channel, and the gateway listens on that channel.

What is a "Last Will"?
When a sensor connects to the gateway, it registers a Last Will message — like writing a note that says "if I suddenly go silent, publish this message for me."

If the sensor loses power or drops off the network without warning, the gateway's MQTT broker automatically publishes the Last Will. This tells AEGIS the sensor went offline unexpectedly, so new readings from that sensor are held until an operator resets the session.

⚠ Known Pitfall — MQTT Client ID Collision on Clone Boards
A common mistake is generating the MQTT client ID by casting ESP.getEfuseMac() to a 32-bit integer:
snprintf(id, sizeof(id), "aegis-%08X", (uint32_t)ESP.getEfuseMac()); // ← BROKEN

The ESP32's MAC address is 48 bits. Casting to 32 bits silently throws away the upper 16 bits — which are the manufacturer bytes that differ between boards from the same factory batch. Clone boards (the most common boards available locally) very often share these upper bytes, so all three boards produce the exact same client ID.

When two MQTT clients connect with the same ID, Mosquitto disconnects the first one. Its Last Will fires, the gateway marks it INTERRUPTED, and it starts a reconnection cycle that kicks the other boards off in turn. The fix is to use the full MAC address as a string:
String mac = WiFi.macAddress(); mac.replace(":", ""); // "A4CF12345678" — always unique
Why is the keep-alive set to 60 seconds?
The keep-alive value tells the broker how often to expect a heartbeat ping from the sensor. The broker will close the connection if it does not hear from the sensor within 1.5× the keep-alive value — so at 60 seconds, the broker waits up to 90 seconds before firing the Last Will.

This headroom is necessary because when WiFi drops and reconnects, the board must re-join the network (up to 20 seconds) before it can send MQTT pings again. A 15-second keep-alive gave only 22.5 seconds of tolerance — not enough for a clean WiFi reconnection, causing boards to appear INTERRUPTED even during normal brief WiFi hiccups.

Why does the sensor's clock matter?
One of AEGIS's main jobs is to catch backdated or replayed data — readings that an attacker might try to inject with a fake old timestamp to make it look like historical data. To do this, the gateway checks that the timestamp inside every reading is close to its own clock. If they differ by more than the allowed threshold (default: 3 seconds), the reading is rejected.
⚠ The ESP32 has no built-in clock. Every time it boots, it starts from zero — it has no idea what time it is until it synchronises over the internet via NTP (Network Time Protocol). If NTP sync fails or drifts, readings will be rejected.

setup() vs loop() — two different NTP calls
The sketches use two different NTP functions deliberately:

In setup() — blocking sync:
setupTimeFromNTP() calls configTime() then waits in a loop until getLocalTime() succeeds (up to 10 seconds). This is safe in setup() because the MQTT connection has not been established yet — there is nothing to time out.

In loop() — non-blocking refresh:
configTime() alone tells the ESP32's internal NTP daemon to re-sync in the background. It returns immediately — client.loop() keeps running, the MQTT keep-alive heartbeats keep firing, and the broker never fires the Last Will.

If you use the blocking version in loop(): WiFi reconnect + blocking NTP can take up to 30 seconds. With a 60-second keep-alive, the broker waits 90 seconds before firing the Last Will — so it survives now. But with the old 15-second keep-alive it caused constant INTERRUPTED events.
Why milliseconds?
The timestamp must be in milliseconds since January 1, 1970 (called Unix time in milliseconds). A typical value looks like 1735084800000.

Important: The standard millis() function on the ESP32 counts milliseconds since boot — not since 1970. That value is useless to the gateway. Always use gettimeofday() after NTP sync.

Also: String(timestamp) silently truncates 64-bit numbers on the ESP32. Always use snprintf(buf, sizeof(buf), "%llu", ts_ms) to convert the timestamp to text safely.
What does "registering a sensor" mean?
Before a sensor can send data, you register it — called enrollment. This is like adding a device to a whitelist. You tell the gateway: "This sensor exists, its name is X, it sends a reading every Y seconds, and its values should be between A and B." The gateway then knows what to expect from that device.
⚠ Enrollment is an administrative step, not a security lock. It does not prove the physical device is genuine — it just tells the gateway what behaviour to expect. Physical security of the sensors is separate.

After registration, what happens first? The sensor will show as WAITING until it sends its first reading. That first reading always passes (no prior reading to compare against) and sets the baseline for all future checks.
If a board is restarted or reflashed: Its seq_num counter resets to 1. The gateway will see this as a sequence gap and reject every subsequent reading with SEQ_GAP. You must go to Operator Tools → Session Reset and reset that node before it can send again. This is normal and expected — a reset confirms to the system that you are aware of the interruption.
How does the gateway decide to accept or reject a reading?
Every reading goes through a series of checks in order. If it fails any one check, it is immediately rejected — the reason is logged, and the rest of the checks are skipped. A reading must pass all checks to be accepted. There is no partial acceptance.

Check 1 — Was this reading sent on time?
The gateway looks at the timestamp inside the reading and compares it to its own clock. If the gap is larger than the allowed threshold (default 3 seconds), the reading is rejected. This catches attackers trying to inject old or pre-made data. It also catches readings with timestamps set in the future. A second sub-check looks at the gap between two consecutive readings arriving at the gateway — if it is wildly different from the expected send interval, that is also flagged.

Check 2 — Is this the next expected reading in the sequence?
Every reading has a counter (seq_num) that must go up by exactly 1 each time. If the counter jumps ahead, it means at least one reading was lost or skipped. If it goes backward or repeats, it is a duplicate or a replay attack. Either way, it is rejected.

Complete ESP32 Sensor Template
Copy this code onto your ESP32. Change the lines marked with ← change this to match your setup. This template includes all fixes — correct client ID generation, 60-second keepalive, and non-blocking NTP in the main loop.
This template matches the sketches deployed on Indoor_Temp_Sensor_1/2/3. The three differences between the sketches are: NODE_ID, MQTT_TOPIC, LWT_TOPIC, and PUBLISH_INTERVAL_MS (5000 / 10000 / 15000). The ESP32-WROOM-32 (Sensor 3) also adds client.setBufferSize(512) in setup. 
#include <WiFi.h>
#include <PubSubClient.h>
#include <DHT.h>
#include <time.h>
#include <sys/time.h>

const char* WIFI_SSID     = "YourSSID";              // ← change
const char* WIFI_PASSWORD = "YourPassword";          // ← change
const char* MQTT_BROKER   = "192.168.x.x";           // ← your Raspberry Pi's IP address
const int   MQTT_PORT     = 1883;
const char* NODE_ID       = "Indoor_Temp_Sensor_1";  // ← change to match enrolled node name
const char* MQTT_TOPIC    = "sensors/Indoor_Temp_Sensor_1";       // ← change
const char* LWT_TOPIC     = "aegis/status/Indoor_Temp_Sensor_1";  // ← change

#define DHTPIN  4
#define DHTTYPE DHT11
DHT dht(DHTPIN, DHTTYPE);

const long  GMT_OFFSET_SEC      = 8 * 3600;  // Philippine Time (UTC+8)
const int   DAYLIGHT_OFFSET_SEC = 0;
const char* NTP_SERVER_1        = "pool.ntp.org";
const char* NTP_SERVER_2        = "time.google.com";

const unsigned long PUBLISH_INTERVAL_MS    = 5000;             // ← change per node
const unsigned long NTP_RESYNC_INTERVAL_MS = 5UL * 60UL * 1000UL;

WiFiClient   espClient;
PubSubClient client(espClient);
unsigned long lastPublishTime = 0;
unsigned long lastNtpSyncTime = 0;
unsigned long seq_num         = 0;

void connectToWiFi() {
  WiFi.begin(WIFI_SSID, WIFI_PASSWORD);
  int r = 0;
  while (WiFi.status() != WL_CONNECTED && r++ < 40) delay(500);
  if (WiFi.status() != WL_CONNECTED) { delay(3000); ESP.restart(); }
}

// Blocking — only call this in setup(), where MQTT is not yet connected.
// Calling it in loop() stalls client.loop() and can trigger the broker's
// Last Will if the block takes longer than 1.5 × keepalive seconds.
void setupTimeFromNTP() {
  configTime(GMT_OFFSET_SEC, DAYLIGHT_OFFSET_SEC, NTP_SERVER_1, NTP_SERVER_2);
  struct tm timeinfo;
  int r = 0;
  while (!getLocalTime(&timeinfo) && r++ < 20) delay(500);
  lastNtpSyncTime = millis();
}

unsigned long long getUnixTimeMs() {
  struct timeval tv;
  gettimeofday(&tv, NULL);
  return ((unsigned long long)tv.tv_sec * 1000ULL)
       + ((unsigned long long)tv.tv_usec / 1000ULL);
}

// Client ID uses the full 12-digit MAC address string.
// DO NOT cast ESP.getEfuseMac() to uint32_t — that drops the upper 2 bytes,
// and clone boards sharing those bytes will produce the same client ID,
// causing each board to kick the others off the broker on connect.
void connectToMQTT() {
  while (!client.connected()) {
    String mac = WiFi.macAddress();  // "A4:CF:12:34:56:78"
    mac.replace(":", "");            // "A4CF12345678"
    String clientId = "aegis-" + mac;
    client.connect(clientId.c_str(), NULL, NULL, LWT_TOPIC, 0, false, "offline");
    if (!client.connected()) delay(2000);
  }
}

bool readDHT11WithRetry(float &temperature, float &humidity) {
  for (int i = 1; i <= 3; i++) {
    humidity    = dht.readHumidity();
    temperature = dht.readTemperature();
    if (!isnan(humidity) && !isnan(temperature)) return true;
    if (i < 3) delay(250);
  }
  return false;
}

String buildPayload(unsigned long long ts_ms, unsigned long seq,
                    float temperature, float humidity) {
  char ts_buf[24], temp_buf[10], hum_buf[10];
  snprintf(ts_buf,   sizeof(ts_buf),   "%llu", ts_ms);
  snprintf(temp_buf, sizeof(temp_buf), "%.2f", temperature);
  snprintf(hum_buf,  sizeof(hum_buf),  "%.2f", humidity);
  String p = "{";
  p += "\"node_id\":\""    + String(NODE_ID) + "\",";
  p += "\"seq_num\":"      + String(seq)      + ",";
  p += "\"timestamp\":"    + String(ts_buf)   + ",";
  p += "\"sensor_value\":" + String(temp_buf) + ",";
  p += "\"humidity\":"     + String(hum_buf);
  p += "}";
  return p;
}

void setup() {
  Serial.begin(115200);
  dht.begin();
  delay(2000);
  connectToWiFi();
  setupTimeFromNTP();         // Blocking sync is safe here — MQTT not connected yet
  client.setServer(MQTT_BROKER, MQTT_PORT);
  client.setKeepAlive(60);   // 60s → broker timeout = 90s, survives WiFi reconnects
  // For ESP32-WROOM-32 only — add this line:
  // client.setBufferSize(512);
}

void loop() {
  unsigned long now = millis();

  // Non-blocking background re-sync — configTime() returns immediately.
  // The NTP daemon handles the actual sync in the background.
  if (now - lastNtpSyncTime >= NTP_RESYNC_INTERVAL_MS) {
    configTime(GMT_OFFSET_SEC, DAYLIGHT_OFFSET_SEC, NTP_SERVER_1, NTP_SERVER_2);
    lastNtpSyncTime = now;
  }

  // On WiFi drop: reconnect, then non-blocking time refresh only.
  if (WiFi.status() != WL_CONNECTED) {
    connectToWiFi();
    configTime(GMT_OFFSET_SEC, DAYLIGHT_OFFSET_SEC, NTP_SERVER_1, NTP_SERVER_2);
    lastNtpSyncTime = now;
  }

  if (!client.connected()) connectToMQTT();
  client.loop();

  if (now - lastPublishTime >= PUBLISH_INTERVAL_MS) {
    lastPublishTime = now;
    float temperature, humidity;
    if (!readDHT11WithRetry(temperature, humidity)) return;
    seq_num++;
    unsigned long long ts_ms = getUnixTimeMs();
    String payload = buildPayload(ts_ms, seq_num, temperature, humidity);
    client.publish(MQTT_TOPIC, payload.c_str());
  }
}
